Identity Governance: From Operations to Architecture

Why I Chose This Project

During my time as a Project Manager and DBA at Deloitte, I managed access requests for an enterprise Data Marketplace. I was responsible for evaluating requests from internal practitioners, global member firms, and third-party vendors. I had to ensure that access provisions adhered to strict licensing agreements, Third-Party Risk Management (TPRM) guidelines, and cross-border data sovereignty laws.

This portfolio page is a two-part showcase. Phase 1 is an interactive simulator demonstrating my operational day-to-day experience managing provisioning workflows. Phase 2 demonstrates my architectural growth: after earning my CompTIA Certification, I learned how to formalize these operational rules into a "Secure by Design" Enterprise RBAC Matrix to prevent Segregation of Duties (SoD) conflicts automatically.

Phase 2: Enterprise RBAC Architecture

The CompTIA Evolution: Relying on human admins to manually toggle checkboxes (like in Phase 1) introduces the risk of human error and insider threats. While studying for my CompTIA certification, I learned the importance of shifting from manual provisioning to "Secure by Design" architecture.

Below is the formal Role-Based Access Control (RBAC) Matrix I designed to automate those business rules. This matrix acts as the absolute source of truth for IAM governance.

Authorized Access Matrix

Enterprise Persona Data Catalog UI Internal US Data Global Data Silo Procurement Contracts
US Practitioner Read Write None Read
Global Member Firm Read None Write None
Third-Party Vendor Read None None None
Platform DBA (IT) Admin Admin Admin None

Audit Findings: Segregation of Duties (SoD)

  • Global Firm + US Data: Giving a Global member firm 'Read' access to US Data creates a cross-border Data Sovereignty violation. The matrix enforces 'None'.
  • Vendor + Procurement: Vendors cannot hold 'Write' or 'Admin' rights in Procurement systems to prevent fraudulent self-contracting.
  • DBA + Procurement: The DBA holds 'Admin' keys to the databases. Giving them 'Write' access to Procurement creates a Toxic Combination where an IT admin could manipulate financial contracts.
